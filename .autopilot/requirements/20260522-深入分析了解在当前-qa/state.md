### Context

**触发**：用户要求评估当前 QA 阶段（SKILL.md L287-426）每个步骤的顺序与并行度是否符合三条工程原则：
1. **基础原则**：约基础的验证越早做（lint/tsc 在前，e2e 在后）
2. **依赖原则**：后续验证依赖前置验证（依赖关系不能并行）
3. **快速原则**：约快的验证越早做（fail-fast，秒级 lint 不应被 60s 构建拖累）

### 不合理点分析（按用户三原则对照）

#### P1 — Tier 3.5 vs Tier 3 依赖关系与"全并行"声明矛盾 ⚠️ 严重

**违反**：依赖原则。

**位置**：SKILL.md L312（"Wave 1 — 命令执行（并行）"）、L329（Tier 3.5 触发条件"Tier 3 已执行（需要 dev server）"）。

**问题**：Wave 1 标题声称所有 Tier 在同一轮 Bash 调用中并行，但 Tier 3.5 依赖 Tier 3 启动的 dev server。同一轮并行启动时，Tier 3.5 会在 Tier 3 dev server 还在 `sleep 8` 时就尝试连接，必然失败或测到旧数据。

**证据**：L329 字面 "Tier 3 已执行（需要 dev server）"；L364 dev server 启动需 `sleep 8` 等待。

**影响**：Tier 3.5 在有性能工具的项目（Lighthouse / size-limit）会随机 ❌，触发自动 ⚠️ 降级（L331），但根因是流程编排错误而非性能问题。

#### P2 — Wave 1 一次性并行所有 Tier 浪费失败快速路径 ⚠️ 中度

**违反**：快速原则、基础原则。

**位置**：L314（"在同一轮响应中发出多个 Bash 工具调用，所有命令独立运行、互不依赖"）+ L340（早期判定门只看 Tier 0+1）。

**问题**：早期判定门只统计 **Tier 0+1** 失败数决定是否跳过 Wave 1.5/2，意味着 Tier 3 dev server 启动（~8s）+ 端点验证、Tier 4 回归扫描的耗时不影响判定，但**已经被启动**。当 Tier 0/1 失败 ≥3 时，Tier 3/3.5/4 的计算和 token 完全是浪费。

**典型场景**：tsc + lint + 红队全 ❌ 时，Tier 3 dev server 已启动且端点已被请求 → 进入 auto-fix 时还要先 kill 这个 server。

**影响**：fail-fast 不彻底，token + 时间双重浪费。

#### P3 — 结果判定步骤 1/2 顺序倒置 ⚠️ 轻度（逻辑瑕疵）

**违反**：依赖原则（步骤 2 是步骤 1 的前提）。

**位置**：L405（步骤 1 场景计数）、L407（步骤 2 格式检查）。

**问题**：步骤 1 要数 `执行:` 标记 → 步骤 2 才验证每个场景是否含 `执行:` + `输出:`。逻辑上：若步骤 2 失败（场景缺标记），步骤 1 的计数就基于不完整数据。正确顺序应该是先验格式再计数，或合并。

**影响**：边界情况（场景有 `执行:` 但缺 `输出:`）下，步骤 1 数值是"假阳性"——计数对了但场景没真正执行完。当前规则虽然在步骤 2 兜底，但顺序倒置易引发 AI 自我合理化（"计数对就过"绕过格式）。

#### P4 — 变更分类被消费两次 ⚠️ 轻度（重复劳动）

**违反**：快速原则（重复扫描浪费时间）。

**位置**：L304-310（前置变更分析：分类）+ L346-354（Wave 1.5 前置：变更类型覆盖检查，对照分类结果）。

**问题**：两段都基于同一份 `git diff` 分类结果做事——前者用于决定影响半径，后者对照得出"应有的 Tier 1.5 场景类型"。完全可以在前置变更分析阶段一次性输出 `{影响半径, 应有场景类型}` 两条结论。

**影响**：AI 多花 1 次 Read/分析，未必造成 bug 但拖慢推进。

#### P5 — Tier 0 "最高优先级"语义模糊 ⚠️ 轻度（文档质量）

**违反**：无原则违反，但与执行模型不符。

**位置**：L316（"Tier 0: 红队验收测试（最高优先级）"）。

**问题**："最高优先级"在执行层面无含义——它和 Tier 1 4 项同轮并行，不会更早完成。语义上是"失败后必须改实现而非改测试"（红队铁律），与执行优先级混淆。

**影响**：易让 AI 错误推断"应该先单独跑 Tier 0 等结果再跑其他"，与同轮并行约定相违。

#### 候选 P6 — 删除 "Wave 1.5" 术语，统一为 "Tier 1.5"（简化）

**观察**：当前文档有两个并行的命名层——"Wave 1 / 1.5 / 2"（执行批次）+ "Tier 0/1/1.5/3/3.5/4"（验证层级），且"Wave 1.5"内部只有"Tier 1.5"一项，形成命名冗余（Wave 1.5 ≡ Tier 1.5）。

**改进**：删除 "Wave 1.5" 术语层。原 L342 标题改为 `#### Tier 1.5 — 真实场景验证（Wave 1 之后、Wave 2 之前）`。文档内所有 "Wave 1.5" → "Tier 1.5"。

**价值**：消除"两个名字指同一件事"的认知负担，对 AI 编排器更友好（减少跨段落引用歧义）。

**成本**：约 8-10 处 string 替换；红队测试若 grep "Wave 1.5" 字面会 fail；[2026-05-16] 决策原文提到"Wave 1.5"需相应更新链接说明。

**推荐度**：⭐⭐⭐⭐（高价值，低成本，但有迁移成本）

#### 候选 P7 — 合并两个"前置"段为单一"启动 QA 前置三步"（简化）

**观察**：当前 L296-310 有两个独立的 "#### 前置：..." 子段（选择性重跑判断 + 变更分析），都属于"在 Wave 1 之前做的事"。L346 又有"##### 前置：变更类型覆盖检查"，三个"前置"散落各处。

**改进**：合并为单一 `#### 启动 QA 前置（三步）`，依次列出：
1. 读 `qa_scope` 决定执行范围
2. 分析变更分类、影响半径、应有场景类型（合并 P4 改进）
3. 触发 stop-hook detect_smoke_eligible（如未触发则维持）

**价值**：单点入口，编排器进入 phase: qa 时只需读一段就知道前置步骤。

**成本**：3 个段合并成 1 个，行数节约 5-8 行；旧测试 grep "前置：变更分析" 字面会 fail。

**推荐度**：⭐⭐⭐（中价值，低成本，与 P4 协同）

#### 候选 P8 — `qa_scope` 三档用表格替代散文（简化）

**观察**：L298-302 用 4 个 bullet（smoke / selective / full / 清除）描述 `qa_scope`，三档差异散落在散文中，阅读时需逐条对比"smoke 跳什么 / selective 跳什么"。

**改进**：用一张三档对照表替代 bullet：

| qa_scope | 触发来源 | 执行范围 | 跳过 |
|----------|---------|---------|------|
| smoke | fast_mode=true 或 stop-hook 检测小 diff | Tier 0/1 + Tier 1.5 + 编排器 3 项内联自审 | qa-reviewer Agent |

**价值**：视觉对比清晰，AI 解析表格优于解析散文。

**成本**：约 4 行 → 7 行（表格头+三行+清除说明），略增；但理解成本下降。

**推荐度**：⭐⭐⭐⭐（高价值，低成本）

#### 候选 P9 — Tier 编号注脚解释"Tier 2 缺失"（小修）

**观察**：Tier 编号 0/1/1.5/3/3.5/4 跳过了 Tier 2，这是因为 [2026-05-07] 决策将原 Tier 2 设计符合性 + Tier 2 代码质量合并为 Wave 2 qa-reviewer，但未释放 Tier 2 编号。新 AI/读者首次进入时会困惑"Tier 2 在哪？"

#### 候选 P10 — 失败清单按文件/根因聚类输出（功能增强，非简化）

### Best Practice 重审（用户反馈后新增）

> 用户反馈："skill 的优化非常难，要非常谨慎小心的优化，非常容易劣化"。本节用 Anthropic Skill Authoring Best Practices 的 5 条原则逐项过 P1-P10，筛掉伪优化。

**5 条原则**：
1. **Concise is key** — 上下文是公共资源，去除已知/冗余信息才算 concise；格式变换不算
2. **Default: Claude is smart** — 改动是否解决"AI 不懂"的问题？还是只让人类阅读更舒服？后者多半伪优化
3. **Consistent terminology** — 改名/合并术语会打破现有术语网络（红队 grep / 历史决策原文引用）
4. **Set appropriate degrees of freedom** — QA 判定是 low-freedom 操作（脆弱、必须按序）；合并简化检查清单 = AI 漏步骤
5. **Avoid over-organizing** — 引入新层级（如 Wave 1a/1b/1c）增加阅读维度

#### 历史 P1 — Tier 3.5 从 Wave 1 并行列表中拆出，明确依赖 Tier 3 完成

#### 修复 P2 — Wave 1 引入"早期判定门"，Tier 3/3.5/4 条件启动

在 Tier 0+1 之后立即判定是否进入 auto-fix，未失败时才启动剩余 Tier。重构为：

- **Wave 1a 基础批（必跑）**：Tier 0 + Tier 1（4 项），同轮 5 个 Bash 调用并行
- **早期判定门**：Wave 1a 完成后立即统计 Tier 0+1 ❌ 数量

- **Wave 1c 依赖批（条件跑）**：Tier 3.5（依赖 Wave 1b 的 Tier 3 dev server）
