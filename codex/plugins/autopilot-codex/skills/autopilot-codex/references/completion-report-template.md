# 完成报告模板

设计原则：**信任来自证据链**（每个声称附可验证证据），**效率来自分层**（一行结论 → 关键数字 → 按需细节），**风险透明**（不隐藏不完美）。

按以下 6 个区块顺序输出，运行 `git log --oneline -1` 和 `git diff --stat HEAD~1` 获取数据：

```markdown
## ✅ autopilot 完成报告

### 结论
{一句话}：{目标描述} → {成功 / 部分成功 / 需关注}

### 关键数字
| 迭代 | 耗时 | 修改文件 | 新增文件 | 新增测试 | QA 通过率 |
|------|------|----------|----------|----------|-----------|
| {N}/{max} | {分钟}min | {N} | {N} | {N} | {通过数}/{总数} |

### 变更清单
（从 `git diff --stat HEAD~1` 提取，按类型分组）

| 文件 | 操作 | 说明 |
|------|------|------|
| `path/to/file.ts` | 修改 | 一句话描述改了什么 |
| `path/to/new.test.ts` | 新增 | 一句话描述测什么 |

### QA 证据链
（从状态文件最后一轮 QA 报告提取，每 Tier 一行）

- **Tier 0 红队验收**: ✅ {N} passed ({耗时}s)
- **Tier 1 基础验证**: ✅ tsc ✅ lint ✅ test({N} passed) ✅ build
- **Tier 1.5 真实场景**: ✅ {场景名}（附关键输出摘录，≤1 行）
- **Tier 2a 设计符合性**: ✅ | **Tier 2b 代码质量**: ✅

### 遗留与风险
（没有则写"无"。诚实列出以下任一项：）
- 降级项：{哪些 Tier 因基础设施不足而跳过或降级}
- 已知限制：{本次实现的边界条件或未覆盖场景}
- 建议：{如有降级 → "建议运行 `/autopilot doctor` 改进工程基础设施"}

### 提交
`{git log --oneline -1 的完整输出}`
```

## 格式要求

- "结论"行不超过 80 字，用户 2 秒内能判断是否需要关注
- "关键数字"用表格，一眼可扫
- "QA 证据链"每 Tier 一行，用 ✅/⚠️/❌ 标记，附关键数字（通过数、耗时）
- "遗留与风险"是信任的关键——主动暴露不完美比被发现更好。无遗留时明确写"无"
- 降级项和工程建议不再单独成步骤，统一收入此区块
