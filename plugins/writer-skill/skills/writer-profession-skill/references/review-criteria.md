# 验收标准：什么样的专业技术文章才是好文章

本文件是 writer-profession-skill 的验收参考，源自团队内部写作标准。

---

## 好文章的公式

好文章 = 好问题 × 好结构 × 好节奏

三者是乘法关系，缺一个就归零。

---

## 好问题：读者为什么要打开这篇文章？

标题和开头的前三句话，决定了 80% 的阅读量。

核心原则：**从问题出发，不从技术出发。** 读者关心的是"我的问题能不能被解决"，而不是"你用了什么技术栈"。

| 标题类型 | 问题 |
|----------|------|
| 基于 Puppeteer 的自动化截图方案 | 读者不知道自己为什么要读 |
| 如何让技术文档中的 SVG 在任何平台都不乱码？ | 读者立刻知道：我也遇到过这个问题！ |

---

## 好结构：读者能不能跟得上？

每一节都是「问题 → 思考 → 解法」的完整闭环。

读者的阅读体验应该是：
> 「原来有这个问题」→「嗯，确实现有方案不行」→「原来可以这样解决，学到了」

而不是：
> 「他在说什么？」→「跟上一段有什么关系？」→「算了不读了」

实操建议：
- 开头给目录（或用自解释标题替代）：让读者有预期，方便跳读
- 每节独立：即使只读某一节，也能有收获
- 阶段递进：用「1.0 → 2.0 → 3.0」串联全文，每个版本解决上个版本的遗留问题

---

## 好节奏：读者愿不愿意读完？

节奏解决可读性问题。同样的内容，排版不同，阅读体验天差地别。

1. **短段落**：每段 3-4 句，超过就拆。大段文字墙是阅读杀手
2. **图文交替**：连续超过 3 段纯文字，就该配一张图、一个表格、或一个代码块
3. **加粗点睛**：每节 1-2 个加粗重点，让「只扫加粗」的读者也能抓住核心
4. **表格承载对比**：涉及对比的内容优先用表格，而非段落描述

---

## 发布前自检清单

- [ ] 标题是否从读者痛点出发？
- [ ] 前三句是否让读者知道能获得什么？
- [ ] 每节是否有完整的「问题 → 思考 → 解法」闭环？
- [ ] 是否有超过 3 段的连续纯文字？
- [ ] 每个加粗是否传递了关键信息？
- [ ] 删掉某一节后文章是否仍然完整？如果是，该节大概率多余
- [ ] 所有数据是否有对比基线和来源？

**核心原则：好文章不是加法，是减法。让不了解背景的读者，花最短的时间，获得最大的收获。**
