# Phase: implement — 红蓝对抗并行实现

> 📋 读取状态文件后，本阶段重点关注 `## 设计文档` 和 `## 实现计划` 区域。

## 目标
通过红蓝对抗模式并行完成编码和验收测试编写。蓝队按计划编码，红队仅基于设计文档编写验收测试，确保测试独立于实现。

## 核心理念
- **信息隔离**：红队只看设计文档，不看蓝队新写的实现代码
- **独立验证**：红队测试验证"应该实现什么"而非"已经实现了什么"
- **并行执行**：蓝队和红队通过 Agent 工具同时启动

> ⚠️ 防合理化：See [references/anti-rationalization.md](references/anti-rationalization.md)

## 工作流程

从状态文件读取 `## 设计文档`。检查是否包含 `## 领域 Skill 委托` 字段：
- **有委托声明** → 走 [1b. Skill 委托路径](#1b-skill-委托路径)
- **无委托声明** → 走 [1a. 蓝/红队对抗路径](#1a-蓝红队对抗路径默认)

### 1a. 蓝/红队对抗路径（默认）

从状态文件读取设计文档和实现计划，**立即**使用 Agent 工具同时启动两个子代理。

#### 蓝队 Agent（实现者）
prompt 参考 `references/blue-team-prompt.md`，填入：
- 实现计划（从状态文件复制）
- 验证方案（从设计文档中复制）
- 项目目录路径和技术栈信息

#### 红队 Agent（验证者）
prompt 参考 `references/red-team-prompt.md`，填入：
- 目标描述和设计文档（**仅**设计，不含实现计划）

**⚠️ 红队铁律**：红队**绝对不能**读取蓝队新写的实现代码。

### 1b. Skill 委托路径

当设计文档声明了 `## 领域 Skill 委托` 时走此路径。

1. **调用领域 Skill**：`Skill: "{skill-name}"`，传递委托输入
2. **收集产出**：`git status` 确认变更，记录到变更日志
3. **补充验收测试（必须执行）**：启动红队 Agent（信息隔离不变）
4. **红队产出检查门**：
   - 有测试文件 → 合流
   - 无测试文件 → 降级为验收检查清单（纯文本）
   - **不允许跳过此步**

**降级**：Skill 失败 → 回退到蓝/红队对抗路径。

### 审查后修改铁律（Post-Review Modification Rule）

**任何在外部审查/评分之后所做的代码修改，必须重新运行对应的验证。**

| 场景 | 要求 |
|------|------|
| 外部 AI 评分后修改代码 | 必须重新评分或重新运行 tsc + 相关测试 |
| 红队测试通过后做"小优化" | 必须重新运行红队测试 |
| Code review 通过后追加改动 | 必须重新运行受影响的 Tier |

### 2. 合流 — 两个 Agent 都完成后

1. 收集蓝队产出：实现摘要、文件列表、困难任务标记
2. 收集红队产出：将测试文件写入项目
3. `git add` 红队的测试文件
4. 更新状态文件：
   - `## 实现计划` 标记 `[x]`
   - 写入 `## 红队验收测试`
   - 追加变更日志
5. 更新 frontmatter：`phase: "qa"`

### 3. 降级策略

- **无测试框架** → 红队产出验收检查清单（纯文本）
- **红队失败** → 记录警告，继续只用蓝队产出
- **蓝队失败** → 设置 `gate: "review-accept"` 等待用户
- **Skill 委托失败** → 回退到蓝/红队对抗
