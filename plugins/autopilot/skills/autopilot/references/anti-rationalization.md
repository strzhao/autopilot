# 防合理化指南（Anti-Rationalization）

> AI 在跳过验证 / 重写测试 / 缩减红队 时常见的自我合理化借口及对应的真相。
> 仅在你想跳过测试 / 重做 / 简化流程时阅读。

## implement 阶段

红蓝对抗模式下，蓝队最常想绕过红队独立验证：

| 借口 | 现实 |
|------|------|
| 太简单 / 先实现再补 | 简单改动也出 bug；后补测试不验证需求 |
| 时间紧跳过 TDD / 红队没必要 | TDD 比 debug 快；自测 = 偏差验偏差 |

要点：信息隔离不是"流程负担"，是测试独立性的保证。红队只看设计文档，蓝队只看实现。打破隔离 = 测试退化为「测我已写的」。

## qa Tier 1.5 真实场景验证

QA 阶段最常见的偷懒是用单元测试结果替代真实场景验证：

| 借口 | 现实 |
|------|------|
| dev server 太重 / 已通过 tsc+jest | `npm run dev &` 等 5 秒即可；单测验证代码结构，真实测试验证用户场景 |
| 设计文档没写 / 后续手动验证 | 没有就自行设计 1 个；QA 阶段就是验证阶段，"后面再验"= 跳过验证 |
| 蓝队已冒烟 / 场景 1 已验核心 | QA 必须独立执行；little-bee-cli 48 测全过但 4 bug 靠手动发现，只跑了 --help |

要点：Tier 1.5 是「功能在真实用户场景下是否可用」，不是「代码是否正确」。每个场景必须有 `执行:` 命令 + `输出:` 真实输出。

## auto-fix 阶段

自动修复阶段最危险的合理化是修改红队测试或盲目改代码：

| 借口 | 现实 |
|------|------|
| 改断言值就过了 / 我知道问题直接修 | 这就是修改红队测试，铁律无例外；70% shotgun fix 引入新 bug，先验证假设再修 |

要点：红队验收测试代表设计意图，失败 = 实现不符合设计，修实现而不是改测试。修复前必须走「观察 → 假设 → 验证 → 修复」四步法。
