# 防合理化指南

AI 编排器倾向于跳过"不方便"的步骤。以下是常见借口及其反驳——在你想跳过某个步骤时，先检查这个表。

## 测试与红队

| 借口 | 现实 |
|------|------|
| "这个改动太简单了，不需要测试" | 简单改动也出 bug。测试只需 30 秒。 |
| "先写实现再补测试" | 后补测试证明的是已写代码能跑，不是需求被满足。 |
| "时间紧，跳过 TDD" | TDD 比 debug 快。跳过是用未来时间换现在的错觉。 |
| "红队没必要，蓝队自己测就行" | 自己测自己 = 用实现偏差验证实现偏差。 |
| "改一下测试的断言值就过了" | 这就是修改红队测试。铁律无例外。 |
| "我知道问题在哪，直接修" | 70% 的 shotgun fix 引入新 bug。先验证假设再修。 |

## Tier 1.5 真实场景

| 借口 | 现实 |
|------|------|
| "需要启动 dev server，太重了" | `npm run dev &` 等 5 秒即可。真实测试的价值 > 启动成本。 |
| "已通过 tsc/jest 验证了" | 单元测试验证代码结构，真实测试验证用户场景。两者不可替代。 |
| "设计文档没写真实测试场景" | 规则明确：没有就自行设计至少 1 个。不是跳过的理由。 |
| "后续会手动验证" | QA 阶段就是验证阶段。"后面再验"= 跳过验证。 |
| "蓝队已经做过冒烟测试了" | 蓝队测试不替代 QA 独立验证。 |
| "场景 1 已验证了核心流程" | --help 验证"能启动"，hanzi list 验证"能用"。 |
