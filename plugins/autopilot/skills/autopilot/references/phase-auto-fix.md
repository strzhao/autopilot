# Phase: auto-fix — 自动修复阶段

> 📋 读取状态文件后，本阶段重点关注 `## QA 报告`（最近一轮）和 `## 红队验收测试` 区域。

## 目标
读取 QA 失败项，逐项分析根因并修复。

## 红队测试铁律
**绝对不允许修改红队验收测试。** 问题在实现，不在测试——无例外。

> ⚠️ 防合理化：See [references/anti-rationalization.md](references/anti-rationalization.md)

## 工作流程

### 1. 读取失败项
从最近一轮 QA 报告中提取所有 ❌ 项目。

### 2. 区分失败来源并确定修复策略

**并行修复**：不同文件且互不依赖的失败项可并行（多个 Edit）。同文件或有依赖时串行。

#### 红队验收测试失败（Tier 0）— 最高优先级
- 实现不符合设计要求 → 修改实现代码
- **禁止修改** `.acceptance.test.*` 文件
- 流程：读测试期望 → 对照设计 → 定位偏差 → 修改实现

#### 蓝队单元测试失败（Tier 1 测试）
- 修复实现中的 bug
- 蓝红测试矛盾时以红队为准

#### 类型/Lint/构建失败（Tier 1 其他）
- 类型错误 → 修正类型声明
- Lint → `eslint --fix` 或手动
- 构建 → 检查导入、依赖、配置

#### 代码质量/安全（Tier 2-4）
- 最小化重构，保持行为不变

#### 真实场景验证失败（Tier 1.5）
- 通常是集成问题（路径、环境、权限、配置）
- 分析实际输出 → 定位偏差 → 修复 → **重新执行场景附证据**

### 3. 逐项修复 — 系统化调试方法论

四阶段：

**a. 观察**：完整阅读错误信息，记录堆栈和位置
**b. 假设**：形成明确因果假设 "X 导致 Y，因为 Z"
**c. 验证**：最小实验验证假设。被推翻则回到观察
**d. 修复**：验证后才修复，`git add` 暂存，运行确认并附证据

### 4. 重试控制
- `retry_count++`
- **< max_retries** → `qa_scope: "selective"`, `phase: "qa"`
  - 例外：Wave 1 快速路径进入的不设 `qa_scope`，全量 QA
- **>= max_retries** → 标注未解决项，`gate: "review-accept"`

### 5. 修复优先级
1. 红队验收测试（Tier 0）
2. 真实场景验证（Tier 1.5）
3. lint/类型错误
4. 蓝队单元测试
5. 构建失败
6. 安全问题
7. 代码质量
