## 工作规则
1. 先扫描项目的测试框架、命名约定和测试文件存放目录
2. 严格按任务列表顺序逐任务实现
2. TDD 模式（项目有测试框架时）：
   - 先写单元测试（RED）—— 描述期望行为
   - **只运行你刚写的单个测试文件**确认失败（`npm test -- <file>` / `pytest <file>`），**不要跑全量套件**（全量留给 QA 阶段统一执行）。**绝不运行 `.acceptance.test.*`**——那是红队验收测试，此刻你的实现很可能未完成、跑它必然全 fail，只会浪费时间并诱导你放宽断言或越界改测试
   - 写最小实现通过测试（GREEN）
   - 重构（REFACTOR）
3. 无测试框架时直接编写实现代码
4. **假设先验证**：当实现依赖对外部系统的假设（API 响应格式、数据结构、字段名）时，先用最小手段验证假设（curl、临时脚本、日志），拿到运行时证据后再编码
5. 每个任务完成后 `git add` 暂存相关文件（不 commit）
6. 单个任务 3 次尝试仍失败，标记为 `[!]` 并继续
7. 不擅自扩大范围，只做计划中的任务
8. **编译期健康自检（交付前）**：所有任务完成后，只做分钟内确定性自检——build（编译通过）/ import（模块可导入）/ 类型检查 / 自写单元测试。证明"我写的代码能编译、能导入、基本跑得起来"。**功能在真实用户场景下是否满足设计意图（真实场景 / 集成 / 调试 / 全量回归）由 QA Tier 1.5 独立验证，蓝队不碰。**
   - **角色边界决策心智**：编译期健康自检之外的任何验证冲动，自问"这是编译期健康自检（我的职责），还是真实场景 / 集成 / 调试（QA Tier 1.5 职责）？"——凡涉及"启动 app 做交互验证 / 反复 bundle 打包 / 全量集成测试 / 深挖运行时卡顿根因"判定为 QA 职责，标 `[!]` 交出，不就地深挖。
   - **终止边界**：编译期健康自检中同一检查反复失败 / 超时 / 挂死，或定位到的问题与本次改动无关（既有 flaky 测试 / 既有运行时问题如 Keychain、GUI 事件循环）→ 标 `[!]` 困难项交 QA，**不无限 retry**。复用规则6 既有 `[!]` 语义，不新增计数器 / 字段。
   - **与规则2 对齐**：编译期健康自检不覆盖全量构建 / 集成测试 / 回归——这些归 QA Tier 1 / 3 / 4。与规则2"全量留给 QA"口径对齐。
9. **引用端点验证**：代码中引用 API 端点路径时（如 manifest 列出 endpoint），必须用 Glob/Grep 验证目标路由文件存在，不能仅从文档推断
10. **契约 1:1 匹配**：实现的接口名 / 字段名 / 边界值运算符 / 错误类型必须与设计文档 `## 契约规约` 章节 1:1 匹配。详情参 references/contract-protocol.md。

## 返回格式
完成后返回：
- 实现摘要（每个任务的完成状态和关键改动）
- 新增/修改的文件列表
- 标记为 [!] 的困难任务（如有）
- 编译期健康自检结果（build / import / 类型检查 / 自写单测的命令与输出，证明代码能编译、能导入、基本跑得起来；真实场景验证归 QA Tier 1.5）
- 发现的设计偏差（如有）
