# 代码审查清单

本清单供 code-quality-reviewer Agent 使用。逐项检查 `git diff` 输出，只标记真实问题。
每个发现必须包含 `file:line` 引用和具体修复建议。无问题的类别直接跳过。

---

## 置信度评分

对每个发现评分 0-100，**只报告 ≥80 的问题**：

| 分数 | 含义 |
|------|------|
| 0-25 | 假阳性或已存在的问题 |
| 26-50 | 小问题，不在项目规范中 |
| 51-75 | 有效但低影响 |
| 76-90 | 重要问题，需要关注 |
| 91-100 | 关键 bug 或明确的规范违反 |

---

## Pass 1 — CRITICAL（最高严重性）

### 安全与数据安全
- SQL 字符串拼接（即使值经过 `.toString()` / `.toInt()` 转换 — 使用参数化查询）
- XSS：不安全的 HTML 渲染（React: `dangerouslySetInnerHTML`、Vue: `v-html`）处理用户可控数据
- 硬编码的密钥、token、密码（应使用环境变量）
- 不安全的反序列化（eval、JSON.parse 无 schema 校验）
- 路径遍历（用户输入直接拼接文件路径）

### 竞态条件与并发
- 读-检查-写无唯一约束或乐观锁
- find-or-create 无数据库唯一索引 — 并发调用可创建重复
- 状态转换未使用原子 `WHERE old_status = ?` — 并发更新可跳过或重复
- 共享可变状态无同步机制

### LLM 输出信任边界
- LLM 生成值（邮箱、URL、名称）写入数据库前未做格式校验
- 结构化工具输出（数组、对象）未做类型/结构检查就直接使用
- LLM 输出直接插入 SQL 或模板引擎

### 枚举与值完整性
- 新增枚举值 → 必须追踪**每一个**消费者（switch/case、if-else、过滤数组、展示逻辑）
- 使用 Grep 查找所有引用，**逐个 READ 验证**，不能只 grep
- 检查 allowlist/filter 数组是否包含新值
- 检查 case/if-elsif 链是否有正确的 default 处理

---

## Pass 2 — INFORMATIONAL（较低严重性）

### 模式一致性
- 代码风格与项目现有代码不一致（命名、缩进、模块结构）
- 导入路径不符合项目约定（绝对 vs 相对、alias 使用）
- 错误处理模式不一致（有的 try-catch，有的 .catch()，有的 Result 类型）

### 边界处理
- 空值/undefined 未处理（可选链滥用隐藏了应该报错的场景）
- 空数组/空字符串未做边界检查
- 超时未设置（HTTP 请求、数据库查询、文件操作）
- 并发边界：Promise.all 无错误隔离（一个失败全部失败）

### 错误处理质量
- 空 catch 块（绝对禁止）
- catch 块只 log 不处理，吞掉了应该向上传播的错误
- 错误信息过于泛化（"发生错误" 而非具体描述）
- fallback 行为未告知用户（静默降级）

### 代码组织
- 单个文件过大（新增代码导致文件显著膨胀）
- 单个函数/方法过长（>50 行应考虑拆分）
- 重复代码（DRY 违反）
- 魔术数字/字符串（应提取为命名常量）

### 测试缺口
- 新增功能无对应测试
- 负向路径测试只检查类型/状态，不检查副作用
- 安全执行路径（鉴权、限流）无集成测试

### 性能
- N+1 查询（循环中的数据库/API 调用，缺少批量加载）
- 视图层 O(n×m) 查找（循环中的 Array.find 应改为 Map/Set）
- 不必要的重渲染（React 组件缺少 memo/useMemo/useCallback）
- 大型列表未虚拟化

### 死代码与一致性
- 赋值但未使用的变量
- 注释/文档描述旧行为，与当前代码矛盾
- 版本号不匹配（PR 标题 vs CHANGELOG vs package.json）

---

## Suppressions — 不应标记的误报

以下模式**不要**标记：
- 冗余但提高可读性的代码（如 `x !== undefined && x !== null` 虽然可简化但更清晰）
- "建议添加注释解释为什么选择这个阈值/常量" — 阈值会调优，注释会腐烂
- 断言已经覆盖了行为，只是"可以更严格"
- 仅为一致性的变更建议（仅因为相邻代码风格不同）
- 正则边界情况 — 当输入已被约束且该情况不会发生
- 测试同时检验多个守卫条件 — 这是正常的
- diff 中已经修复的问题 — 先完整阅读 diff 再评论
- 预存在的问题 — 不属于本次变更引入的

## 输出格式

```
代码质量审查: N 个问题 (X critical, Y important, Z minor)

### Strengths
[具体做得好的地方，带 file:line 引用]

### Issues

#### Critical (必须修复) — 置信度 ≥90
[bug、安全问题、数据丢失风险]

#### Important (应该修复) — 置信度 80-89
[架构问题、缺失功能、错误处理不足、测试缺口]

#### Minor (可以改进) — 置信度 80-89 低影响
[代码风格、优化机会、文档改进]

每个问题格式:
- **[置信度分] 问题标题**
  - 文件: path/to/file:line
  - 问题: 具体描述
  - 影响: 为什么重要
  - 修复: 具体修复方案

### Assessment
Ready to merge: Yes / No / With fixes
Reasoning: [1-2 句技术评估]
```
